# Захист Лабораторної роботи №3: Робота з SQLite у Flutter

Цей документ містить пояснення структури та логіки коду для захисту лабораторної роботи.

## 1. Мета роботи
Розробка мобільного застосунку на Flutter, що використовує локальну базу даних SQLite для зберігання та управління даними користувачів та їх паролів.

## 2. Основні технології та бібліотеки
- **Flutter**: Фреймворк для розробки інтерфейсу.
- **sqflite**: Основна бібліотека для роботи з SQLite на Android та iOS.
- **sqflite_common_ffi**: Використовується для підтримки SQLite на Desktop (Windows), оскільки стандартний `sqflite` працює лише на мобільних ОС.
- **path**: Для формування коректних шляхів до файлів бази даних.

## 3. Структура бази даних (Схема)
База даних `lab3_v2.db` містить дві таблиці:
1. **users**:
   - `id`: Унікальний ідентифікатор (PRIMARY KEY AUTOINCREMENT).
   - `name`: Ім'я користувача (TEXT).
   - `age`: Вік користувача (INTEGER).
2. **passwords**:
   - `id`: Унікальний ідентифікатор.
   - `user_id`: Зовнішній ключ, що вказує на користувача (INTEGER).
   - `service_name`: Назва сервісу (Google, Facebook тощо).
   - `username`: Логін у сервісі.
   - `password_value`: Сам пароль.

## 4. Пояснення коду (lib/main.dart)

### Ініціалізація бази даних (`getDatabase`)
Функція використовує патерн Singleton (`globalDb != null`). Якщо база ще не відкрита, вона створює файл БД та виконує SQL-запити на створення таблиць у колбеку `onCreate`.

### Головна функція (`main`)
Додано перевірку на платформу (Windows/Linux/MacOS). Якщо застосунок запущено на Desktop, ініціалізується FFI-фабрика (`databaseFactoryFfi`), що дозволяє SQLite працювати поза мобільним середовищем.

### Екран користувачів (`UsersScreen`)
- **initState**: Викликає `initDb()`, щоб підключитися до бази при запуску.
- **loadUsers**: Виконує запит `db.query('users')` та оновлює стан через `setState`.
- **addUser**: Збирає дані з текстових полів та робить `db.insert`.
- **deleteUser**: Виконує каскадне видалення: спочатку видаляє всі паролі користувача, а потім самого користувача.
- **showEditDialog**: Відображає діалогове вікно для редагування існуючих записів через `db.update`.

### Екран паролів (`PasswordsScreen`)
- Приймає `userId` як аргумент для фільтрації паролів конкретного користувача.
- **loadPasswords**: Використовує параметри `where` та `whereArgs` для безпечного пошуку паролів: `where: 'user_id = ?'`.
- **toggleVisibility**: Реалізує приховування/показ паролів через список `visibleIds`, що зберігає ID записів, які зараз відкриті.

## 5. Можливі питання на захисті
1. **Як працює `setState`?**
   - Це метод, що повідомляє Flutter, що стан змінився, і потрібно перемалювати віджет з новими даними.
2. **Чому ми використовуємо `?` у запитах (`where: 'id = ?'`)?**
   - Це захист від SQL-ін'єкцій. Значення підставляються бібліотекою окремо від тексту запиту.
3. **Навіщо потрібен `sqflite_common_ffi`?**
   - Стандартний `sqflite` покладається на нативні канали зв'язку Android/iOS. На Windows таких каналів немає, тому ми використовуємо прямий запуск SQLite через інтерфейс сторонніх функцій (FFI).
4. **Як реалізовано зв'язок між таблицями?**
   - Через поле `user_id` у таблиці `passwords`, яке відповідає `id` з таблиці `users`. Це зв'язок "один до багатьох".
